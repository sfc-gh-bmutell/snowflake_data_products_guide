Discovery & Catalog

Horizon Catalog provides a unified repository for discovering and understanding all data assets in your Snowflake environment. It combines AI-powered search, automatic metadata enrichment, and organizational data product discovery.

Universal Search

Natural language search powered by Snowflake Cortex AI enables users to find data assets without knowing exact object names or locations. Search across databases, schemas, tables, views, and columns using business terms.

  • AI-powered semantic search across all Snowflake objects
  • Search by business terms, not just technical names
  • Filter by object type, owner, tags, and more
  • Cross-account and cross-region discovery

AI-Powered Object Descriptions

Automatically generate descriptions for tables, views, and columns using Snowflake Cortex. This helps users understand data context without manual documentation.

-- Enable AI-generated descriptions for a table
ALTER TABLE my_db.my_schema.customers
  SET AI_DESCRIPTION = TRUE;

-- View the generated description
DESCRIBE TABLE my_db.my_schema.customers;

Internal Marketplace

The Internal Marketplace enables teams to discover and access organizational data products (listings) without data duplication. Data stewards can curate and publish governed datasets for internal consumption.

  • Browse organizational listings shared within your company
  • Request access to data products with approval workflows
  • No data copying ‚Äî access data in place
  • Full RBAC integration for access control

Object Insights

Understand data popularity, access patterns, quality metrics, and dependencies through the Object Insights interface in Snowsight.

  • Popularity: See which tables and columns are most frequently queried
  • Access Patterns: Understand who is using your data and how
  • Quality Metrics: View data quality scores from DMFs
  • Dependencies: Visualize upstream and downstream lineage

Classification & Compliance

Snowflake provides powerful tools for classifying data and ensuring compliance with regulations like GDPR, HIPAA, and CCPA. Object tagging and automatic classification enable governance at scale.

Object Tagging

Tags are key-value pairs that you attach to Snowflake objects (databases, schemas, tables, columns, roles, users) to enable classification, cost tracking, and policy enforcement.

Tag Category Example Values Use Case
SensitivityPII, PHI, CONFIDENTIAL, PUBLICDrive masking policies and access controls
DomainFINANCE, MARKETING, HR, OPERATIONSEnable domain-based governance and ownership
Quality TierGOLD, SILVER, BRONZEIndicate data product reliability for consumers
LifecyclePRODUCTION, STAGING, DEPRECATEDControl pipeline access and retention
 
-- Create a tag for PII classification
CREATE TAG governance.tags.pii_classification
  ALLOWED_VALUES 'HIGHLY_SENSITIVE', 'SENSITIVE', 'PUBLIC'
  COMMENT = 'Classifies data sensitivity level';

-- Apply tag to a column
ALTER TABLE customers MODIFY COLUMN email
  SET TAG governance.tags.pii_classification = 'SENSITIVE';

-- Query tagged objects
SELECT * FROM snowflake.account_usage.tag_references
WHERE tag_name = 'PII_CLASSIFICATION';

Automatic Sensitive Data Classification

Snowflake's classification engine automatically scans column metadata and sample data to identify PII and sensitive attributes, then applies system tags for governance.

-- Create a classification profile with auto-tagging
CREATE OR REPLACE SNOWFLAKE.DATA_PRIVACY.CLASSIFICATION_PROFILE
  my_classification_profile(
  {
    'minimum_object_age_for_classification_days': 0,
    'maximum_classification_validity_days': 30,
    'auto_tag': true,
    'classify_views': true,
    'tag_map': {
      'column_tag_map': [
        {
          'tag_name': 'governance.tags.pii',
          'tag_value': 'highly_sensitive',
          'semantic_categories': ['NAME', 'NATIONAL_IDENTIFIER']
        },
        {
          'tag_name': 'governance.tags.pii',
          'tag_value': 'sensitive',
          'semantic_categories': ['EMAIL', 'PHONE_NUMBER']
        }
      ]
    }
  });

-- Apply classification profile to a schema
ALTER SCHEMA my_db.my_schema
  SET CLASSIFICATION_PROFILE = 'my_classification_profile';

-- Manually classify a specific table
CALL SYSTEM$CLASSIFY('my_db.my_schema.customers', 
  {'classification_profile': 'my_classification_profile'});

Security & Access Control

Snowflake Horizon provides multiple layers of security through Role-Based Access Control (RBAC), dynamic data masking, and row access policies. These controls enable fine-grained data protection without data duplication.

Role-Based Access Control (RBAC)

RBAC is the foundation of Snowflake security. Permissions are granted to roles, and users are assigned to roles. This enables scalable security management through role hierarchies.

-- Create a role hierarchy for data access
CREATE ROLE data_reader;
CREATE ROLE data_analyst;
CREATE ROLE data_engineer;

-- Build hierarchy: engineer > analyst > reader
GRANT ROLE data_reader TO ROLE data_analyst;
GRANT ROLE data_analyst TO ROLE data_engineer;

-- Grant permissions to base role
GRANT USAGE ON DATABASE analytics TO ROLE data_reader;
GRANT USAGE ON SCHEMA analytics.public TO ROLE data_reader;
GRANT SELECT ON ALL TABLES IN SCHEMA analytics.public TO ROLE data_reader;

Dynamic Data Masking

Masking policies control what users see at the column level based on their role. The actual data never changes ‚Äî Snowflake applies transformations at query time.

-- Create a masking policy for SSN
CREATE MASKING POLICY mask_ssn AS (val STRING) 
  RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('HR_ADMIN', 'COMPLIANCE_OFFICER') THEN val
    ELSE 'XXX-XX-' || RIGHT(val, 4)
  END;

-- Apply masking policy to a column
ALTER TABLE employees MODIFY COLUMN ssn
  SET MASKING POLICY mask_ssn;

-- Create a masking policy for email
CREATE MASKING POLICY mask_email AS (val STRING)
  RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('MARKETING_ADMIN') THEN val
    ELSE REGEXP_REPLACE(val, '.+\@', '***@')
  END;

Tag-Based Masking

The most scalable approach: bind masking policies to tags, then tag columns. The policy automatically applies to any column with that tag.

-- Create a tag for PII
CREATE TAG governance.tags.pii;

-- Create masking policies for different data types
CREATE MASKING POLICY mask_pii_string AS (val STRING)
  RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('DATA_OWNER', 'COMPLIANCE_ADMIN') THEN val
    ELSE '**MASKED**'
  END;

-- Bind masking policy to the tag
ALTER TAG governance.tags.pii
  SET MASKING POLICY mask_pii_string;

-- Now any column tagged with 'pii' is automatically protected
ALTER TABLE customers MODIFY COLUMN email
  SET TAG governance.tags.pii = 'true';

ALTER TABLE customers MODIFY COLUMN phone
  SET TAG governance.tags.pii = 'true';

Row Access Policies

Row access policies filter entire rows based on the querying role or user attributes. Use mapping tables for flexible, maintainable access rules.

-- Create a mapping table for role-to-region access
CREATE TABLE governance.role_region_mapping (
  role_name STRING,
  allowed_region STRING
);

INSERT INTO governance.role_region_mapping VALUES
  ('SALES_MANAGER_EMEA', 'EMEA'),
  ('SALES_MANAGER_APAC', 'APAC'),
  ('SALES_MANAGER_AMER', 'AMER'),
  ('EXECUTIVE', 'EMEA'),
  ('EXECUTIVE', 'APAC'),
  ('EXECUTIVE', 'AMER');

-- Create row access policy
CREATE ROW ACCESS POLICY region_filter AS (region_col STRING)
  RETURNS BOOLEAN ->
  EXISTS (
    SELECT 1 FROM governance.role_region_mapping
    WHERE role_name = CURRENT_ROLE()
      AND allowed_region = region_col
  );

-- Apply row access policy to table
ALTER TABLE sales_data
  ADD ROW ACCESS POLICY region_filter ON (region);

Best Practice Policy Lifecycle

Always test policies in development before production. Poorly designed policies can block legitimate access or create performance issues. Use the POLICY_REFERENCES Information Schema function to verify policy assignments.

Data Quality Monitoring

Data Metric Functions (DMFs) enable continuous monitoring of data quality within Snowflake. Use system-provided DMFs for common checks or create custom DMFs for business-specific validation.

System Data Metric Functions

Snowflake provides built-in DMFs in the SNOWFLAKE.CORE schema:

Category DMF Description
AccuracyNULL_COUNTCount of NULL values in a column
NULL_PERCENTPercentage of NULL values
BLANK_COUNTCount of blank/empty values
BLANK_PERCENTPercentage of blank values
FreshnessFRESHNESSData freshness based on timestamp column or last DML
UniquenessDUPLICATE_COUNTNumber of duplicate values
UNIQUE_COUNTNumber of unique non-NULL values
VolumeROW_COUNTTotal number of records
ValidityACCEPTED_VALUESCheck if values match expected criteria

Associating DMFs with Tables

-- Associate NULL_COUNT DMF with a column
ALTER TABLE customers
  ADD DATA METRIC FUNCTION SNOWFLAKE.CORE.NULL_COUNT
    ON (email);

-- Associate ROW_COUNT DMF with a table
ALTER TABLE orders
  ADD DATA METRIC FUNCTION SNOWFLAKE.CORE.ROW_COUNT
    ON ();

-- Check for freshness based on timestamp column
ALTER TABLE events
  ADD DATA METRIC FUNCTION SNOWFLAKE.CORE.FRESHNESS
    ON (event_timestamp);

-- Validate accepted values
ALTER TABLE orders
  ADD DATA METRIC FUNCTION SNOWFLAKE.CORE.ACCEPTED_VALUES
    ON (status, status -> status IN ('PENDING', 'SHIPPED', 'DELIVERED'));

Setting DMF Schedules

-- Set DMF to run every 60 minutes
ALTER TABLE customers
  SET DATA_METRIC_SCHEDULE = '60 MINUTES';

-- Set DMF to run on a cron schedule (daily at 6 AM)
ALTER TABLE customers
  SET DATA_METRIC_SCHEDULE = 'USING CRON 0 6 * * * America/Los_Angeles';

-- Set DMF to run when table is modified
ALTER TABLE customers
  SET DATA_METRIC_SCHEDULE = 'TRIGGER_ON_CHANGES';

Creating Custom DMFs

-- Create a custom DMF for referential integrity
CREATE DATA METRIC FUNCTION governance.dmfs.referential_check(
  arg_t1 TABLE (arg_c1 INT),
  arg_t2 TABLE (arg_c2 INT)
)
RETURNS NUMBER AS
$$
  SELECT COUNT(*) FROM arg_t1
  WHERE arg_c1 NOT IN (SELECT arg_c2 FROM arg_t2)
$$;

-- Apply custom DMF to validate foreign key
ALTER TABLE orders
  ADD DATA METRIC FUNCTION governance.dmfs.referential_check
    ON (customer_id, TABLE(my_db.public.customers(id)));

Viewing DMF Results

-- Query DMF results from the monitoring view
SELECT
  measurement_time,
  metric_database,
  metric_schema,
  metric_name,
  table_name,
  column_names,
  value
FROM snowflake.account_usage.data_quality_monitoring_results
WHERE measurement_time > DATEADD(day, -7, CURRENT_TIMESTAMP())
ORDER BY measurement_time DESC;

Creating Data Products

Key Section Data Products for Data Stewards

This section covers how data governance managers and stewards can create, publish, and manage data products using Snowflake's Internal Marketplace and Provider Studio.

What is a Data Product?

A data product is a curated, governed dataset or application published as a listing. Data products enable self-service data access while maintaining governance controls. They can be shared internally (organizational listings) or externally (Marketplace listings).

Listing Type Audience Use Case
Organizational ListingsInternal teams within your organizationShare curated datasets across departments without data duplication
Private ListingsSpecific external Snowflake accountsShare data with partners, customers, or subsidiaries
Marketplace ListingsAll Snowflake customersMonetize data or provide public datasets

Step 1: Prepare Your Data

Before creating a data product, ensure your data is properly governed:

  • Apply appropriate tags for classification (PII, sensitivity level, domain)
  • Set up masking policies for sensitive columns
  • Configure row access policies if needed
  • Associate DMFs to monitor data quality
  • Document the data with descriptions and metadata

Step 2: Create a Provider Profile

A provider profile is required to publish listings (except for free internal listings).

-- Grant privileges to create listings
USE ROLE ACCOUNTADMIN;
GRANT CREATE DATA EXCHANGE LISTING ON ACCOUNT TO ROLE data_steward_role;

-- Navigate to Provider Studio in Snowsight:
-- Data Products ‚Üí Provider Studio ‚Üí Profiles ‚Üí + Profile

Provider Profile Setup (Snowsight UI)

  1. Sign in to Snowsight with ACCOUNTADMIN or a role with CREATE DATA EXCHANGE LISTING privilege
  2. Navigate to Data Products ‚Üí Provider Studio
  3. Select Profiles tab, then click + Profile
  4. Fill in organization name, contact information, and description
  5. Submit for verification (required for Marketplace listings)

Step 3: Create an Organizational Listing (Internal)

Organizational listings are shared within your Snowflake organization through the Internal Marketplace.

Creating an Organizational Listing (Snowsight UI)

  1. Navigate to Data Sharing ‚Üí Internal Sharing
  2. Click + Listing
  3. Enter a descriptive name for your listing
  4. Under "Who can discover," select Anyone in your organization
  5. Click + Select to attach database objects (tables, views, secure views)
  6. Add a description explaining the data, its source, and intended use
  7. Optionally add sample queries or documentation
  8. Click Publish
-- Create a secure view for your data product
CREATE SECURE VIEW analytics.data_products.customer_360 AS
SELECT
  c.customer_id,
  c.customer_segment,
  c.region,
  o.total_orders,
  o.total_revenue,
  o.last_order_date
FROM core.customers c
JOIN (
  SELECT
    customer_id,
    COUNT(*) AS total_orders,
    SUM(amount) AS total_revenue,
    MAX(order_date) AS last_order_date
  FROM core.orders
  GROUP BY customer_id
) o ON c.customer_id = o.customer_id;

-- Add description for discoverability
COMMENT ON VIEW analytics.data_products.customer_360 IS
  'Customer 360 view combining customer demographics with order history. 
   Updated daily. Contact: data-team@company.com';

Step 4: Create a Private Listing (External)

Share data with specific external Snowflake accounts (partners, customers).

Creating a Private Listing

  1. Navigate to Data Products ‚Üí Provider Studio
  2. Click + Listing
  3. Enter a name and select Only specified consumers
  4. Attach your database objects
  5. Add consumer Snowflake account identifiers (org_name.account_name)
  6. Set access type: Free, Limited Trial, or Paid
  7. Add terms of use and documentation
  8. Click Publish

Step 5: Publish to Snowflake Marketplace (Public)

For publicly available data products, publish to the Snowflake Marketplace.

Marketplace Publication Process

  1. Complete and verify your Provider Profile
  2. Create a listing in Provider Studio with Anyone on the Marketplace
  3. Add comprehensive documentation, sample queries, and data dictionary
  4. Configure pricing (Free, Limited Trial, or Paid)
  5. Submit for Snowflake review and approval
  6. Once approved, click Publish to make it available globally

Best Practices for Data Stewards

Practice Description
